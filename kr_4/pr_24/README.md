## Описание проекта
Данная работа представляет собой комплексное микросервисное приложение, объединяющее результаты практических заданий №19–23. В проекте реализована работа с базами данных, механизмы кэширования, балансировка нагрузки и контейнеризация всех компонентов.

## Реализованный функционал

### 1. Работа с базами данных (Практики №19, №20)
* Реализовано серверное приложение с использованием **PostgreSQL** для хранения реляционных данных о пользователях.
* Интегрирована NoSQL база данных **MongoDB** для работы с неструктурированными документами пользователей.
* Для обеих систем реализован полный набор CRUD-эндпоинтов (`GET`, `POST`, `PATCH`, `DELETE`).

### 2. Кэширование с Redis (Практика №21)
* Внедрена система кэширования для оптимизации часто запрашиваемых данных.
* Настроено время жизни кэша (TTL): 1 минута для списка пользователей и 10 минут для товаров.
* Реализована инвалидация (очистка) кэша при обновлении или удалении данных.

### 3. Балансировка нагрузки и отказоустойчивость (Практика №22)
* В качестве балансировщика настроен **Nginx**, распределяющий трафик между несколькими backend-серверами.
* Использованы параметры `max_fails` и `fail_timeout` для автоматического исключения упавших серверов из цепочки обработки запросов.

### 4. Контейнеризация (Практика №23)
* Созданы **Dockerfiles** для сборки образов backend-сервисов.
* Описан файл **docker-compose.yml**, объединяющий бэкенд, базы данных, Redis и Nginx в единую изолированную сеть `bridge`.

## Стек технологий
* **Runtime:** Node.js (Express)
* **DBs:** PostgreSQL, MongoDB
* **Cache:** Redis
* **Proxy/Load Balancer:** Nginx
* **Orchestration:** Docker Compose

## Инструкция по запуску

Проект запускается в среде **WSL (Ubuntu)**.

1. **Сборка и запуск контейнеров:**
   Выполните команду ниже. Она соберет образы и запустит все сервисы в фоновом режиме:
   `docker compose up --build -d`

2. **Проверка работы балансировщика:**
   Выполните несколько последовательных запросов к API:
   `curl http://localhost/`
   Ответы должны приходить от разных серверов (например, `backend-1`, `backend-2`) по алгоритму Round Robin.

3. **Проверка отказоустойчивости:**
   Остановите один из контейнеров:
   `docker compose stop backend1`
   При повторном запросе Nginx должен перенаправить трафик на оставшиеся рабочие узлы без прерывания обслуживания.

## Структура проекта
* `/backend` — исходный код сервера и Dockerfile.
* `/nginx` — конфигурационные файлы балансировщика (`nginx.conf`).
* `docker-compose.yml` — описание всей инфраструктуры.
